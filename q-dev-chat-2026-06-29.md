Project Overview

The application is divided into three major modules:

1. Master Data Configuration
2. Manual Transactional Module
3. Production Dashboard

The application is used inside a manufacturing plant to configure master data, record daily production transactions manually, and visualize production KPIs.

---

Module 1 – Master Data Configuration

This module is used only by administrators to configure all prerequisite master data.

There are 5 sections distributed across 2 screens.

General Rules:

- Every section has its own ADD and SAVE buttons.
- Tables are editable.
- Clicking ADD inserts a new blank row.
- Only one row can be edited at a time.
- Tables should support scrolling or pagination.
- Saving one section should not affect the others.

Section 1 – Machine Layout

Purpose:
Configure the manufacturing hierarchy.

Columns:

- Plant
- Work Centre
- Work Station
- Module

Relationship:

Plant
→ Work Centre
→ Work Station
→ Module

Dropdowns should follow this hierarchy.

---

Section 2 – Reason Code Definition

Purpose:
Configure all production reason codes.

Columns:

- Reason Code
- Description
- Category
- Reason Type

Category values:

- Machine Downtime
- NOK

Reason Type (only for Machine Downtime):

- Organizational Downtime (TO)
- Technical Downtime (TT)
- Maintenance Time (TW)

---

Section 3 – Variant Definition

Purpose:
Configure products.

Columns:

- Material Number
- Material Description
- Material Traceability Level

---

Section 4 – Shift Definition

Columns:

- Shift Name
- Shift Duration
- Break Time

Rules:

- Every shift name must be unique.
- One shift can contain multiple breaks.

---

Section 5 – Shift Planning

Columns:

- Shift
- Work Centre
- Active

Rules:

- Multiple shifts can be assigned to one Work Centre.
- Total active shift duration cannot exceed 24 hours.

---

Module 2 – Manual Transactional Module

Single screen.

Sections:

1. Header
2. Downtime &amp; OEE
3. Production Data
4. Customer Complaint &amp; Accident

Initially only Header is expanded.

All other sections are collapsed.

---

Header

Fields:

- Plant
- Work Centre
- Shift
- Date

Buttons:

- SHOW
- EDIT

Rules:

- All fields mandatory.
- Plant → Work Centre dependency.
- Work Centre → Shift dependency.
- SHOW loads saved records.
- EDIT enables editing.

---

Downtime &amp; OEE

Contains three subsections.

Machine Downtime

Fields:

- Module
- Reason Code
- Duration

Buttons:

- ADD
- SAVE

Table:

- Module
- Reason Code
- Duration

---

Machine Target Cycle

Fields:

- Module
- Target Cycle Time

Buttons:

- ADD
- SAVE

---

Resource Planning

Fields:

- Work Station
- Duration
- Resource Count
- Resource Names

Buttons:

- ADD
- SAVE

Rules:

- Tables are read-only until EDIT is clicked.
- Single-row editing only.

---

Production Data

Fields:

- Variant Type
- OK Count
- NOK Count
- NOK Reason Code
- NOK Type

NOK Type:

- Scrap
- Rework
- Retest

Validation:

- OK and NOK accept numeric values only.

Buttons:

- ADD

import React, { createContext, useContext, useState } from 'react';

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  const login = (employeeId, role) => {
    setUser({ employeeId, role });
  };

  const logout = () => {

  );
}

const BASE_URL = 'http://localhost:5000/api';

const getHeaders = () => ({
  'Content-Type': 'application/json',
  Authorization: `Bearer ${localStorage.getItem('token')}`,
});

export const api = {
  get: (url) => fetch(`${BASE_URL}${url}`, { headers: getHeaders() }).then(r => r.json()),
  post: (url, body) => fetch(`${BASE_URL}${url}`, { method: 'POST', headers: getHeaders(), body: JSON.stringify(body) }).then(r => r.json()),
  put: (url, body) => fetch(`${BASE_URL}${url}`, { method: 'PUT', headers: getHeaders(), body: JSON.stringify(body) }).then(r => r.json()),
  delete: (url) => fetch(`${BASE_URL}${url}`, { method: 'DELETE', headers: getHeaders() }).then(r => r.json()),
};
